# Qing Zai Development Guide

This guide describes how to set up, run, and validate the Qing Zai Flutter project locally.

## Prerequisites

- Flutter 3.x on the stable channel.
- Dart 3.x, usually bundled with Flutter.
- A local editor with Flutter and Dart plugins.
- Platform SDKs for the targets you want to build:
  - Android Studio and Android SDK for Android.
  - Xcode and CocoaPods for iOS.
  - Visual Studio Desktop development with C++ workload for Windows.
  - A validated Flutter-HarmonyOS toolchain before HarmonyOS platform folders are generated.

## Initial setup

```bash
flutter pub get
flutter analyze
flutter test
```

If platform folders are missing in a fresh clone, generate the supported Flutter platform shells first:

```bash
flutter create . --platforms=android,ios,windows
```

HarmonyOS shells are intentionally not generated by default. Add them only after the project has a confirmed Flutter-HarmonyOS SDK/toolchain.

## Recommended local workflow

1. Pull the latest `main`.
2. Run `flutter pub get` after dependency changes.
3. Implement code in a small vertical slice.
4. Add or update tests next to the changed layer.
5. Run:

```bash
flutter analyze
flutter test
```

6. Update README or docs when behavior, setup, or deployment changes.

## Project layers

- `lib/app`: application entry and navigation shell.
- `lib/core`: shared models, services, security, offline support, sync, and providers.
- `lib/features`: feature-specific models, business providers, and UI.
- `test/core`: core service, sync, security, offline, and performance tests.
- `test/features`: model and business-provider tests.

## Security development rules

- Never log raw passwords, WebDAV secrets, Authorization headers, API keys, tokens, or URLs containing embedded credentials.
- Use `SensitiveValueRedactor` before showing caught errors in UI.
- Initialize encryption through `EncryptionService.initialize(password)` so `PasswordPolicy` is enforced.
- Keep WebDAV paths relative to the configured server root.
- Require SHA-256 checksums for self-hosted update packages.

## Sync development rules

- Syncable models must preserve `id`, `deviceId`, `updatedAt`, `isDeleted`, and `lamportClock`.
- Deletions should be represented as tombstones instead of hard deletes when data may sync.
- Conflict resolution must remain deterministic across devices.
- Remote sync payloads are gzip-compressed before encryption, but old uncompressed payloads must remain readable.

## Manual smoke test checklist

Use this checklist before a release candidate:

- Create, update, complete, and delete a todo.
- Add a clipboard item and toggle favorite.
- Create, search, pin, and delete a note.
- Create a memo with an attachment path.
- Start and complete Pomodoro sessions.
- Configure WebDAV and run a sync.
- Disable network, create local changes, then restore network and flush the offline queue.
- Start LAN receive mode and perform a loopback/manual file transfer.
- Check update manifest handling on a self-hosted HTTPS manifest.
